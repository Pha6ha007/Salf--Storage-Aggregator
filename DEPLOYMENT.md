# Production Deployment Guide

Complete guide for deploying StorageCompare.ae to production using **Vercel** (frontend) and **Railway** (backend).

## 📋 Overview

- **Frontend**: Next.js 14 → Vercel
- **Backend**: NestJS → Railway
- **Database**: PostgreSQL → Railway
- **Cache**: Redis → Railway
- **Domain**: storagecompare.ae (configure DNS later)

## 🚀 Deployment Architecture

```
┌─────────────────────────────────────────────┐
│  Vercel (Frontend)                          │
│  https://storagecompare.vercel.app          │
└────────────────┬────────────────────────────┘
                 │ HTTPS
                 ▼
┌─────────────────────────────────────────────┐
│  Railway (Backend API)                      │
│  https://api-storagecompare.up.railway.app │
└─────┬───────────────────────┬───────────────┘
      │                       │
      ▼                       ▼
┌─────────────┐         ┌──────────────┐
│ PostgreSQL  │         │    Redis     │
│  (Railway)  │         │  (Railway)   │
└─────────────┘         └──────────────┘
```

## 🔧 Part 1: Backend Deployment (Railway)

### Prerequisites

1. **GitHub Account** with repository pushed
2. **Railway Account**: [railway.app](https://railway.app)
3. **API Keys** ready (Google Maps, Anthropic, Twilio, SendGrid, etc.)

### Step 1: Create Railway Project

1. Go to [railway.app](https://railway.app)
2. Click **"New Project"**
3. Select **"Deploy from GitHub repo"**
4. Choose your repository
5. Select the **backend** directory: `/src/backend`

### Step 2: Add PostgreSQL Database

1. In your Railway project, click **"+ New"**
2. Select **"Database" → "PostgreSQL"**
3. Railway will auto-generate `DATABASE_URL`
4. Add **PostGIS** extension:
   - Click on PostgreSQL service
   - Go to **"Data"** tab
   - Run SQL:
   ```sql
   CREATE EXTENSION IF NOT EXISTS "postgis";
   CREATE EXTENSION IF NOT EXISTS "vector";
   CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
   ```

### Step 3: Add Redis

1. Click **"+ New"** again
2. Select **"Database" → "Redis"**
3. Railway will auto-generate `REDIS_URL`

### Step 4: Configure Environment Variables

In Railway backend service, go to **"Variables"** tab and add:

#### Required Variables

```bash
# Node Environment
NODE_ENV=production
PORT=${{PORT}}  # Railway provides this automatically

# API
API_PREFIX=api/v1

# Frontend URL (will update after Vercel deployment)
FRONTEND_URL=https://storagecompare.vercel.app

# Database (auto-generated by Railway)
DATABASE_URL=${{Postgres.DATABASE_URL}}

# Redis (auto-generated by Railway)
REDIS_URL=${{Redis.REDIS_URL}}

# JWT (IMPORTANT: Generate secure secrets!)
JWT_SECRET=<generate-with-openssl-rand-base64-32>
JWT_ACCESS_EXPIRY=15m
JWT_REFRESH_EXPIRY=7d

# Google Maps
GOOGLE_MAPS_API_KEY=<your-google-maps-key>

# AWS S3
AWS_REGION=me-south-1
AWS_S3_BUCKET=storagecompare-media-production
AWS_ACCESS_KEY_ID=<your-aws-key>
AWS_SECRET_ACCESS_KEY=<your-aws-secret>

# SendGrid
SENDGRID_API_KEY=<your-sendgrid-key>
SENDGRID_FROM=noreply@storagecompare.ae

# Twilio
TWILIO_ACCOUNT_SID=<your-twilio-sid>
TWILIO_AUTH_TOKEN=<your-twilio-token>
TWILIO_PHONE_NUMBER=+971xxxxxxxxx
TWILIO_WHATSAPP_NUMBER=whatsapp:+971xxxxxxxxx

# Anthropic Claude
ANTHROPIC_API_KEY=<your-anthropic-key>
ANTHROPIC_MODEL=claude-sonnet-4-20250514

# Paddle (optional for MVP)
PADDLE_API_KEY=<your-paddle-key>
PADDLE_WEBHOOK_SECRET=<your-paddle-webhook-secret>
PADDLE_ENVIRONMENT=production
PADDLE_SELLER_ID=<your-seller-id>

# Sentry (error monitoring — strongly recommended in production)
SENTRY_DSN=<your-sentry-dsn>
APP_VERSION=1.0.0
```

#### Generate Secure Secrets

```bash
# Generate JWT_SECRET
openssl rand -base64 32

# Generate JWT_REFRESH_SECRET (if needed)
openssl rand -base64 32
```

### Step 5: Deploy Backend

1. Railway will auto-deploy after you push to GitHub
2. Monitor deployment in **"Deployments"** tab
3. Check logs for any errors

### Step 6: Run Database Migrations

Railway automatically runs migrations via `Procfile`, but if needed manually:

1. Go to Railway backend service
2. Click **"Settings" → "Deploy"**
3. Custom Start Command: `npx prisma migrate deploy && node dist/main.js`

### Step 7: Seed Production Database (Optional)

Only run once on first deployment:

```bash
# In Railway backend service console
npm run seed:production
```

This creates:
- System admin user
- Default CRM activity types

**⚠️ Change admin password immediately after first login!**

### Step 8: Get Backend URL

1. Go to **"Settings" → "Networking"**
2. Click **"Generate Domain"**
3. You'll get: `https://api-storagecompare.up.railway.app`
4. **Copy this URL** - needed for Vercel deployment

### Step 9: Test Backend

```bash
curl https://api-storagecompare.up.railway.app/api/v1/health

# Expected response:
{
  "status": "ok",
  "timestamp": "2026-03-02T..."
}
```

## 🎨 Part 2: Frontend Deployment (Vercel)

### Prerequisites

1. **Vercel Account**: [vercel.com](https://vercel.com)
2. **Backend URL** from Railway (previous step)

### Step 1: Import Project to Vercel

1. Go to [vercel.com](https://vercel.com)
2. Click **"Add New..." → "Project"**
3. **Import** your GitHub repository
4. **Configure**:
   - Framework Preset: **Next.js**
   - Root Directory: **frontend**
   - Build Command: `npm run build`
   - Output Directory: `.next`

### Step 2: Configure Environment Variables

In Vercel project settings, go to **"Settings" → "Environment Variables"**:

```bash
# Backend API URL (from Railway)
NEXT_PUBLIC_API_URL=https://api-storagecompare.up.railway.app/api/v1

# Site URL (will be Vercel URL initially)
NEXT_PUBLIC_SITE_URL=https://storagecompare.vercel.app

# Google Maps
NEXT_PUBLIC_GOOGLE_MAPS_API_KEY=<your-google-maps-key>

# Environment
NODE_ENV=production
```

**⚠️ Important**: All `NEXT_PUBLIC_*` variables are exposed to the browser. Never put secrets in them!

### Step 3: Deploy Frontend

1. Click **"Deploy"**
2. Vercel will build and deploy automatically
3. Monitor deployment progress

### Step 4: Get Frontend URL

After deployment:
1. You'll get: `https://storagecompare.vercel.app`
2. **Update Railway backend** `FRONTEND_URL` variable with this URL
3. Redeploy Railway backend for CORS to take effect

### Step 5: Test Frontend

1. Visit `https://storagecompare.vercel.app`
2. Test search, catalog, authentication
3. Check browser console for errors

## 🌐 Part 3: Custom Domain Setup (Optional)

### Configure Custom Domain on Vercel

1. In Vercel project → **"Settings" → "Domains"**
2. Add **storagecompare.ae** and **www.storagecompare.ae**
3. Configure DNS at your domain registrar:

```
# Add these DNS records:
CNAME  www   cname.vercel-dns.com
A      @     76.76.21.21  # Vercel IP (check Vercel docs for current IP)
```

### Configure API Subdomain on Railway

1. In Railway backend → **"Settings" → "Networking"**
2. Add custom domain: **api.storagecompare.ae**
3. Configure DNS:

```
CNAME  api   <your-railway-subdomain>.up.railway.app
```

### Update Environment Variables

After custom domains are configured:

**Vercel:**
```bash
NEXT_PUBLIC_API_URL=https://api.storagecompare.ae/api/v1
NEXT_PUBLIC_SITE_URL=https://storagecompare.ae
```

**Railway:**
```bash
FRONTEND_URL=https://storagecompare.ae
```

## 🔄 Part 6: Rollback Procedures

### Rollback Frontend (Vercel)

1. Go to **"Deployments"** tab
2. Find previous working deployment
3. Click **"⋯" → "Promote to Production"**

### Rollback Backend (Railway)

1. Go to **"Deployments"** tab
2. Find previous working deployment
3. Click **"Redeploy"**

### Database Rollback

⚠️ **Database rollbacks are tricky!** Best practices:

1. **Before risky migrations**:
   ```bash
   # Create backup
   Railway → PostgreSQL → "Settings" → "Create Backup"
   ```

2. **Restore from backup**:
   - Contact Railway support OR
   - Use `pg_dump` / `pg_restore`

## 🔐 Part 7: Security

### Что уже реализовано в коде

| Уровень | Защита |
|---|---|
| **HTTP заголовки** | Helmet: CSP, HSTS, X-Frame-Options, noSniff, Referrer-Policy |
| **Аутентификация** | JWT в httpOnly cookies, sameSite:strict, refresh rotation |
| **CORS** | Строгий allowlist, только конкретные домены |
| **Rate limiting** | Глобально 100/мин, auth 5/мин, refresh 10/мин |
| **Валидация** | ValidationPipe whitelist — лишние поля отрезаются |
| **БД** | Prisma параметризованные запросы — SQL injection исключён |
| **Ошибки** | 500-е не содержат stack trace в production |
| **Swagger** | Отключён в production — API карта не публична |
| **Seed** | Закрыт за admin role — не публичный эндпоинт |
| **Sentry** | Мониторинг 500-х, cookies/passwords автоматически скрыты |

### Обязательные шаги после деплоя

#### 1. Включить GitHub Security Settings

В репозитории → **Settings → Security**:
- ☑ Secret scanning
- ☑ Push protection (блокирует push если в коммите найден API ключ)
- ☑ Dependabot alerts
- ☑ Dependabot security updates

#### 2. Настроить Sentry

```bash
# 1. Зарегистрироваться на sentry.io (бесплатно до 5000 ошибок/мес)
# 2. Создать проект: Node.js → "storagecompare-backend"
# 3. Скопировать DSN

# Добавить в Railway Variables:
SENTRY_DSN=https://your-key@sentry.io/your-project-id
APP_VERSION=1.0.0
```

#### 3. Проверить security headers

```bash
# Должны присутствовать все заголовки
curl -I https://api.storagecompare.ae/api/v1/health

# Ожидаемые заголовки:
# X-Content-Type-Options: nosniff
# X-Frame-Options: DENY
# Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
# X-DNS-Prefetch-Control: off
# Referrer-Policy: strict-origin-when-cross-origin
```

#### 4. Убедиться что Swagger закрыт

```bash
# Должен вернуть 404 в production
curl https://api.storagecompare.ae/api/v1/docs
```

#### 5. Сменить пароль admin

После первого деплоя — зайти в приложение под admin и сменить пароль через профиль.

#### 6. Сгенерировать сильный JWT_SECRET

```bash
# Минимум 32 символа
openssl rand -base64 48
```

### Автоматическое сканирование CI/CD

Workflow `security-scanning.yml` запускается при каждом PR и push в `main`:

```
CodeQL       — статический анализ кода (injection, XSS, path traversal)
TruffleHog   — поиск реальных утечек API ключей в git history  
npm audit    — CVE в production зависимостях (блокирует на high/critical)
Pattern scan — regex поиск хардкода AWS/Anthropic/Twilio ключей
─────────────────────────────────────────────────────────────────
Security Gate — все checks должны пройти перед merge в main
```

**Dependabot** каждый понедельник создаёт PR с патчами безопасности.

### Ротация ключей (раз в 90 дней)

```bash
# Список ключей для ротации:
# - JWT_SECRET          → openssl rand -base64 48
# - AWS_ACCESS_KEY_ID   → AWS Console → IAM → Create new key
# - ANTHROPIC_API_KEY   → console.anthropic.com
# - SENDGRID_API_KEY    → app.sendgrid.com
# - TWILIO_AUTH_TOKEN   → console.twilio.com

# После смены JWT_SECRET все активные сессии инвалидируются
# (пользователи выйдут из системы — предупредить заранее)
```

## 🆘 Part 9: Troubleshooting

### Common Issues

#### Frontend can't connect to backend

**Symptom**: Network errors, CORS errors

**Fix**:
1. Check `NEXT_PUBLIC_API_URL` in Vercel
2. Check `FRONTEND_URL` in Railway
3. Verify CORS configuration in `main.ts`
4. Check Railway service is running

#### Database connection errors

**Symptom**: `Error: P1001: Can't reach database`

**Fix**:
1. Check `DATABASE_URL` in Railway
2. Verify PostgreSQL service is running
3. Check IP whitelist (Railway has none by default)
4. Run migrations: `npx prisma migrate deploy`

#### Build failures on Vercel

**Symptom**: Build fails with TypeScript errors

**Fix**:
1. Run `npm run build` locally first
2. Check all environment variables are set
3. Verify `next.config.mjs` is correct
4. Check build logs for specific errors

#### Migrations fail on Railway

**Symptom**: Deployment succeeds but app crashes

**Fix**:
1. Check migration files in `prisma/migrations`
2. Run migrations manually in Railway console
3. Check PostgreSQL extensions are installed
4. Verify `DATABASE_URL` format
