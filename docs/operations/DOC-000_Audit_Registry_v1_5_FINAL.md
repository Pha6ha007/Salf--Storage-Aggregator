## 1. Purpose of This Registry

This document serves as the **single authoritative source of truth** for the status, version, and audit history of all project documentation.

### What This Registry Does:

- Lists all project documents (DOC-001 through DOC-108)
- Records canonical versions for each document
- Tracks audit status and audit history
- Provides a centralized view of documentation completeness
- Maintains governance rules for document lifecycle

### What This Registry Does NOT Do:

- Does NOT describe system architecture, features, or implementation
- Does NOT replace or summarize the content of other documents
- Does NOT introduce new statuses, phases, or requirements
- Does NOT make assumptions about unaudited documents
- Does NOT manage product roadmap or release planning

This registry is a **control document**, not a content document.

## 2. Audit Status Legend

The following statuses are used to classify document audit state:

| Status | Symbol | Meaning |
|--------|--------|---------|
| **GREEN** | 🟢 | Document has been audited and is canonical. No changes required. |
| **YELLOW** | 🟡 | Document has been audited but requires corrections or minor updates. |
| **RED** | 🔴 | Document has been audited and is not acceptable. Major rework required. |
| **NOT AUDITED** | ⚪ | Document exists but has not been formally reviewed. |
| **NOT AVAILABLE** | ⚫ | Document has not been created yet. |

**Note:** Only documents with 🟢 GREEN status are considered canonical and ready for implementation.

## 6. Governance Rules

### 6.1. Status Change Authority

Only the following roles can change document status:

| Action | Required Approval |
|--------|------------------|
| Mark as 🟢 GREEN | Technical Leadership + Product Owner |
| Mark as 🟡 SUPPORTING | Technical Lead + 1 Senior Engineer |
| Mark as 🔴 RED | Technical Leadership |
| Mark as ⚪ NOT AUDITED | Document Author |
| Mark as ⚫ NOT AVAILABLE | Product Owner |

### 6.2. Canonical Document Rules

A document can ONLY be marked 🟢 GREEN if:

1. ✅ It has been technically reviewed by at least 2 senior engineers
2. ✅ It is consistent with DOC-001 (Functional Specification)
3. ✅ All cross-references to other documents are accurate
4. ✅ Naming conventions match across API, database, and architecture docs
5. ✅ It does not contradict any existing 🟢 GREEN document
6. ✅ It has been approved by Technical Leadership

### 6.3. Version Control

- All 🟢 GREEN documents must follow semantic versioning: `v{major}.{minor}`
- Major version changes require re-audit
- Minor version changes require technical review only
